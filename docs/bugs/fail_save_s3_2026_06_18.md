# Failed to save data to S3 (2026-06-18)

## Context

- Study iteration: `mirrorview_scaled_2026_06_18`
- S3 bucket: `jspsych-mirror-view-4`
- Save endpoint: `POST /save-jspsych-data` → Lambda `jspsych-scroll-save-data`
- Frontend error after completing a full run:

```text
Save request failed (500): {"error":"Failed to save data to S3"}
```

## Problem

Participant data never reached S3. The study completed in the browser but the final save step returned HTTP 500.

## Diagnosis

**Mismatch between deployed Lambda code and IAM/env config.**

| Layer | Expected (scaled run) | Actually deployed |
|-------|----------------------|-------------------|
| Lambda code `BUCKET_NAME` | `jspsych-mirror-view-4` (env or default) | Hardcoded `jspsych-mirror-view-3` |
| IAM `s3:PutObject` | `jspsych-mirror-view-4/data/*` | ✓ already updated by Terraform |
| Lambda env `BUCKET_NAME` | `jspsych-mirror-view-4` | ✓ already set by Terraform |

Terraform updated IAM and environment variables when switching buckets, but **did not redeploy the Lambda zip** because `source_code_hash` was unchanged. Stale JS kept writing to bucket-3 → `AccessDenied`.

CloudWatch log pattern:

```text
Error saving data to S3: AccessDenied: User ... is not authorized to perform: s3:PutObject on resource: arn:aws:s3:::jspsych-mirror-view-3/data/test/...
```

Local code (fixed, not yet deployed at time of bug):

```javascript
const BUCKET_NAME = process.env.BUCKET_NAME || "jspsych-mirror-view-4";
```

Old deployed code hardcoded:

```javascript
const BUCKET_NAME = "jspsych-mirror-view-3";
```

## Solution

1. **Force-redeploy save-data Lambda** — a plain `terraform apply` may report "no changes" while AWS still runs stale code (state/code hash drift). Use `-replace`:

```bash
cd infra
terraform apply \
  -var-file=../jobs/terraform/mirrorview_scaled_2026_06_18.tfvars \
  -replace=aws_lambda_function.save_data \
  -auto-approve
```

2. **Recreate API Gateway invoke permission** (destroy/recreate drops it):

```bash
terraform apply \
  -var-file=../jobs/terraform/mirrorview_scaled_2026_06_18.tfvars \
  -auto-approve
```

3. **Verify deployment:**

```bash
# Env vars
aws lambda get-function-configuration \
  --function-name jspsych-scroll-save-data \
  --query 'Environment.Variables'

# Direct invoke (test payload)
aws lambda invoke \
  --function-name jspsych-scroll-save-data \
  --payload '{"httpMethod":"POST","body":"{\"csv\":\"col\\n1\",\"prolific_id\":\"manual-test-save-verify\"}"}' \
  /tmp/save-test.json && cat /tmp/save-test.json

# Confirm object in bucket-4
aws s3 ls s3://jspsych-mirror-view-4/data/test/ --recursive | tail -5
```

4. **Re-run study** with a fresh PID and confirm save succeeds (HTTP 200, object under `data/test/` or `data/prolific/`).

**Prevention:** Lambda code changes require zip rebuild + deploy. If env/IAM updated but saves still fail, compare `aws lambda get-function` `CodeSha256` to `terraform state show data.archive_file.save_data_zip` `output_base64sha256`. After `-replace`, always re-apply to restore `aws_lambda_permission`.
